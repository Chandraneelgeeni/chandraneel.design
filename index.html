How I work

Design process

Every project starts with understanding the system — not the screen.

01
Ecosystem mapping
Before wireframes, I map every touchpoint — all platforms, user roles, data flows, and integration dependencies — to understand the full system I'm designing within.
→
02
Domain research
I research the technical and regulatory domain deeply before touching UI. Understanding MCCB systems or DER regulations shapes the UX decisions more than any user interview.
→
03
Information architecture
Structure before aesthetics. I define hierarchy, content models, and navigation logic before any visual work — because bad IA can't be fixed with good design.
→
04
System design
I build component-driven design systems — tokens, components, variants, dark/light modes — before designing screens. Scale is a systems problem, not a visual one.
→
05
Collaborative refinement
I work directly with engineers to validate design decisions against implementation constraints, ensuring handoffs produce working products — not just beautiful specs.

The problem

A technically capable product with a broken experience

When I joined Blaze Automation, the B.One Plus product existed across multiple disconnected surfaces — a consumer app, a B2B portal, and several internal dashboards — with no unified design language, inconsistent interaction patterns, and no dark mode despite the product living on screens 24/7.

The onboarding flow dropped users before device setup completed. Automation creation required 14+ steps for a basic trigger-action sequence. Device management screens used five different button styles across the app. The B2B portal shared zero visual language with the consumer app despite serving overlapping users. The product was technically capable but experientially fragmented.

Complexity handled

Designing for two fundamentally different audiences on one platform

The same physical device — a smart switch — needed a simple toggle for a homeowner and a real-time energy audit event log for a utility operator. These are incompatible mental models on the same platform. The design system had to serve both without forcing either.
  • Consumer UX and enterprise UX coexisting in the same ecosystem with divergent information needs and interaction expectations
  • 250+ screen scope requiring token-level design system architecture before any individual screen work could scale reliably
  • Retrofitting dark mode across an existing light-mode product — requiring full color audit and redesign of every semantic decision
  • Real-time IoT constraints: device offline states, Zigbee/BLE connectivity failures, and delayed state sync all needed explicit UX affordances
  • Four major third-party integrations each carrying separate technical APIs, compliance requirements, and user mental models
Systems thinking

Mapping the ecosystem before designing the screens

Before any screen work, I mapped the full product ecosystem — all user roles, touchpoints, data flows, and integration relationships. This revealed the critical architectural gap: the consumer app and B2B portal were designed as separate products, not as two views of the same system. Users who crossed both contexts (common for ADR enrollees) experienced jarring visual and interaction discontinuity.

Key UX decisions

Decisions that defined the product

Decision 01 — System
Token-first design system before any screen work
Alternative: screen-first → extract tokens later
With 250+ screens across light and dark modes, building screens before establishing the token system would have created exponential rework. Tokens first paid back within the first two weeks.
Decision 02 — Automation UX
Progressive disclosure for automation creation
Alternative: single-screen wizard with all fields
IoT automations involve conditions, device states, timing logic, and override rules. A single-screen wizard collapsed under this complexity. Progressive disclosure showed users exactly what they needed at each decision point.
Decision 03 — ADR
Consent-forward enrollment with always-visible override
Alternative: opt-out enrollment for higher adoption
Utility-controlled device management is inherently trust-sensitive. Opt-in consent with a permanent, high-visibility manual override built the user trust that made sustained enrollment possible. Short-term adoption isn't worth long-term disengagement.
Decision 04 — DER
Abstract regulatory terminology behind plain-language UI
Alternative: mirror compliance language directly in UI
DER regulations are written for utility engineers, not homeowners. Translating compliance requirements into conversational UX required reading both the regulation and the user's mental model simultaneously — compliance and usability are a translation problem.
Sub-integration · ADR

Automated Demand Response — two audiences, one coherent system

ADR is the utility-industry mechanism for managing electricity grid stability by temporarily adjusting household appliance behavior during peak demand periods. The UX challenge was simultaneously serving two audiences with opposing needs: utility operators who need real-time grid event visibility, device compliance data, and bulk control tools, and consumers who need to understand what's happening to their devices, why, and how to take back control.

The central UX problem: how do you communicate a utility-triggered device event — technically complex, potentially disruptive — in a way that feels informative rather than alarming? I designed a notification and status layer that translated grid event language ("DR event initiated — residential load shed 18%") into consumer language ("Your AC is on energy-saving mode until 6PM. Tap to override."). The utility sees the technical view. The consumer sees the human one. Both views are generated from the same event data.

Sub-integration · DER

Distributed Energy Resources — when compliance shapes UX

DER integration brought solar systems, EV batteries, and household stored energy into the ADR ecosystem — introducing both regulatory compliance requirements and bidirectional energy flow complexity. My research process here was unconventional: I read government energy regulation documents alongside user interviews, mapping the legal requirements to user mental models simultaneously.

Understanding regulatory intent behind each compliance step allowed me to design enrollment flows that satisfied legal requirements without surfacing technical language in the UI. Compliance and usability don't conflict — they require translation. The compliance form doesn't have to say "DER enrollment agreement pursuant to CERC regulation 6.4" — it can say "Your solar system will help balance the grid. Here's what that means for you."

Sub-integration · Video Doorbell

Smart device integration — getting the interaction architecture right

The Video Doorbell integration required designing the full device interaction architecture: live feed access, motion event notification flow, two-way audio initiating from both the app and the doorbell, visitor history management, and privacy mode toggles. I led competitive analysis across Ring, Nest Hello, and Arlo before designing the interaction model.

A critical production decision: the motion-triggered notification animations needed to communicate urgency without being alarming. I designed the Lottie/JSON animation workflow to ensure the notification motion was lightweight (<30KB), semantically appropriate, and consistent across both Android and iOS rendering environments.

Reflection

What I'd do differently

Learnings
"If I rebuilt this ecosystem, I'd establish the token system before designing the first component — not retrofitting dark mode onto a light-mode system. We built the architecture backwards, which was expensive. The design system decision is an architecture decision. It belongs at the start, not after the screens exist. Token-first would have halved the dark mode workload and made the design system genuinely scalable rather than retroactively organized."

The challenge

Turning industrial data density into operational clarity

Industrial energy management platforms carry enormous data loads — real-time consumption metrics, tariff schedules, EV charge session logs, device health signals, maintenance alert queues, and factory-floor monitoring — all competing for screen space and cognitive attention. The design challenge wasn't surfacing the data. It was deciding what to show, when, to whom — and what to make invisible.

The platform served multiple distinct user roles with fundamentally different information needs: facility managers needing operational overview, energy analysts needing time-series consumption data, EV fleet operators needing charge session billing data, and maintenance engineers needing equipment health status. A single dashboard layout served none of them well.

Architecture approach

Widgetized, role-configurable dashboard architecture

I designed the platform around composable, modular dashboard widgets — independent data panels that different user roles could configure around their specific monitoring priorities. This shifted the dashboard from a one-size-fits-all report view to a configurable operational workspace where the information hierarchy adapts to the person, not the product.

  • Energy insights module: time-series consumption with anomaly flagging, comparative baselines, and plain-language summaries adjacent to charts
  • EV billing management: charge session logs, tariff tier tracking, billing reconciliation workflows, and port utilization heatmaps
  • Maintenance module: equipment health status dashboard, preventive scheduling, fault history, alert escalation workflows
  • Factory monitoring: industrial load profiles, circuit-level current monitoring, demand spike detection with automatic alert generation
  • Tariff management: rate schedule visualization, real-time tariff tracking, peak/off-peak period indicators
Key UX insight
Industrial operators don't have time to interpret data visualizations under pressure. Every chart needed a plain-language machine summary adjacent to it — not "consumption peaked at 247kWh at 14:32" but "High consumption event — 23% above baseline. Check HVAC zone 3." The interpretation layer is part of the design.
Decision

Progressive alert severity — not everything is an emergency

An early finding: the existing alert system treated every notification with identical visual weight. Operators were experiencing alert fatigue — critical fault conditions were buried beneath routine status updates. I redesigned the alert architecture around three severity tiers with distinct visual, audio, and interaction affordances. Critical alerts could not be passively dismissed — they required explicit acknowledgment with a logged reason. Medium alerts surfaced in the notification stream. Low priority alerts accumulated silently in a log view, accessible but not intrusive.

Reflection
Learnings
"Enterprise dashboard design is 80% information architecture and 20% visual design. The hardest decisions weren't about color or chart type — they were about hierarchy: what does a facility manager need in the first 10 seconds of opening this dashboard? That single question drove every design decision. If you can't answer it, the dashboard isn't finished."

Context

Designing within an existing system — without breaking it

BeHome247 came with an established design language and a support team with deeply ingrained workflows. My role here wasn't to redesign — it was to extend and improve within existing constraints. This required understanding the system deeply before touching it: auditing existing component patterns, mapping the support agent's full diagnostic workflow, and identifying exactly where friction occurred.

The core finding: support agents were context-switching across six or more screens to diagnose a single hub issue — moving between device status, connectivity logs, event history, ticket management, and device commands on separate screens with no shared context. I redesigned the diagnostic flow around a hub-centric unified view that consolidated device status, connectivity timeline, recent events, and available actions onto a single surface.

The design challenge

Reducing diagnostic time without redesigning the system

  • Adapting to an existing component library — every new pattern had to be buildable with existing components without creating visual inconsistency
  • Designing for expert users who had learned workarounds in the existing system — changes couldn't break muscle memory without payback
  • Surfacing the right diagnostic signals — not all IoT event data is useful for support; separating noise from signal was a core UX problem
  • Balancing information density with cognitive load — support agents handle multiple tickets simultaneously and need fast context restoration
Core insight
Support UX is about speed, not aesthetics. The measure of success isn't how the dashboard looks — it's how quickly an agent can move from 'ticket received' to 'root cause identified.' Every design decision was evaluated against that time constraint first, and everything else second.
Reflection
Learnings
"Working within an existing system taught me discipline. Constraints are clarifying — when you can't redesign everything, you sharpen your judgment about what actually matters to the user's core task. The change that saves 45 seconds per ticket across 200 daily tickets matters far more than ten visual refinements. Constraint focuses design on outcomes rather than aesthetics."

The domain

Understanding the system deeply before designing for it

MCCB (Moulded Case Circuit Breakers) and ACB (Air Circuit Breakers) are the electrical control infrastructure inside industrial buildings — managing high-current distribution, overload protection, and load switching for entire floors or facilities. A factory engineer managing these systems needs real-time current load visibility, temperature monitoring, trip event diagnostics, and predictive maintenance signals — all accessible instantly, from a single platform.

Before designing a single screen, I spent time understanding the fault response workflow: what engineers checked first when an alert fired, what the critical decision path was from "anomaly detected" to "circuit isolated," and what information was genuinely needed versus what was system noise. In industrial environments, domain understanding precedes design decisions — entirely.

Design approach

Operational visibility for high-stakes, time-pressured environments

Industrial monitoring interfaces carry a specific usability requirement that consumer products don't: they must communicate critical status clearly, at distance, under pressure, and often while the operator is simultaneously doing physical work. I designed the monitoring views around a scan-first hierarchy — color-coded device status legible from across a control room, critical numeric readouts scaled for arm's-length reading, and alert escalation that required explicit acknowledgment rather than passive dismissal.

  • Analytics views: load profiles, peak demand events, and consumption baselines over configurable time windows — with plain-language anomaly summaries adjacent to every chart
  • Maintenance workflow: preventive scheduling, fault event history with root cause tagging, component lifecycle tracking, and service record management
  • Trip event diagnostic flow: from alert detection to root cause identification to reset action in three steps maximum — no branching menus under pressure
  • Alert architecture: three severity tiers — critical (requires explicit acknowledgment), warning (surfaces prominently), informational (accessible in log, not intrusive)
Key UX decision
I designed the fault alert to include a plain-language interpretation alongside every technical reading: "94A load — 94% of 100A trip rating. Risk of thermal overload if sustained above 15 minutes." Not just the number. The number plus what it means. Industrial operators know the systems — they need the interface to remove uncertainty, not add it.
Reflection
Learnings
"Industrial UX demanded a different kind of empathy. The user isn't at a desk calmly evaluating options — they're standing in a control room with an alarm active, under time pressure, with physical safety implications. Designing for that context forced me to ruthlessly simplify every interaction path. If a critical action took more than three steps, that was a design failure. Not a tradeoff — a failure."
